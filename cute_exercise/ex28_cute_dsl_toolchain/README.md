# How do I inspect the embedded CuTe DSL CUDA compiler?

## Question

Given a Python environment with `nvidia-cutlass-dsl` installed, how do we answer
these toolchain questions?

- Does the package ship a standalone `ptxas` binary, or is the assembler path
  embedded somewhere else?
- Which CUDA compiler / `ptxas` version is used?
- Which NVVM version is used?
- Which PTX ISA version does CuTe DSL emit?
- How do we prove whether CuTe DSL shells out to an external assembler during
  compilation?

## Setup

Use the `cute-exercise` conda environment from the root README. The concrete
package investigated here was:

```text
nvidia-cutlass-dsl[cu13]==4.4.2
```

The test machine was an NVIDIA GB200 system. Torch reported compute capability
`(10, 0)`, so the CuTe DSL compile path targeted Blackwell.

## Commands

Find the native CuTe DSL MLIR extension:

```bash
SO="$CONDA_PREFIX/lib/python3.12/site-packages/nvidia_cutlass_dsl/python_packages/cutlass/_mlir/_mlir_libs/_cutlass_ir.cpython-312-aarch64-linux-gnu.so"
```

Collect static package evidence:

```bash
file "$SO"
readelf -d "$SO" | rg 'NEEDED|RPATH|RUNPATH'
ldd "$SO"
strings "$SO" | rg -i 'Cuda compilation tools|NVIDIA NVVM version|LLVM version|CL-[0-9]+|nvPTXCompiler|ptxas'
```

Compile one CuTe DSL kernel and dump PTX/CUBIN:

```bash
rm -rf /tmp/cute_dsl_probe
mkdir -p /tmp/cute_dsl_probe
CUTE_DSL_KEEP_PTX=1 CUTE_DSL_KEEP_CUBIN=1 CUTE_DSL_DUMP_DIR=/tmp/cute_dsl_probe \
  python - <<'PY'
import torch
import cutlass.cute as cute
from cutlass.cute.runtime import from_dlpack
from cute_exercise.ex1_for_loop.kernel import ElementwiseAdd

M, N = 64, 512
a = torch.randn(M, N, device="cuda", dtype=torch.float16)
b = torch.randn(M, N, device="cuda", dtype=torch.float16)
c = torch.empty_like(a)

compiled = cute.compile(
    ElementwiseAdd("vectorized"),
    from_dlpack(a, assumed_align=16),
    from_dlpack(b, assumed_align=16),
    from_dlpack(c, assumed_align=16),
)

print(compiled.__ptx__.splitlines()[:12])
print("cubin bytes:", len(compiled.__cubin__))
PY
```

Inspect the emitted artifacts:

```bash
sed -n '1,16p' /tmp/cute_dsl_probe/*.ptx
cuobjdump --dump-elf /tmp/cute_dsl_probe/*.cubin | rg -A12 '.note.nv.tkinfo|.note.nv.cuinfo'
```

Trace the compile to check for external compiler loads or process launches:

```bash
rm -rf /tmp/cute_dsl_probe
mkdir -p /tmp/cute_dsl_probe
CUTE_DSL_KEEP_PTX=1 CUTE_DSL_KEEP_CUBIN=1 CUTE_DSL_DUMP_DIR=/tmp/cute_dsl_probe \
strace -f -o /tmp/cute_dsl_probe/trace.log -e trace=execve,openat \
  python - <<'PY'
import torch
import cutlass.cute as cute
from cutlass.cute.runtime import from_dlpack
from cute_exercise.ex1_for_loop.kernel import ElementwiseAdd

M, N = 64, 512
a = torch.randn(M, N, device="cuda", dtype=torch.float16)
b = torch.randn(M, N, device="cuda", dtype=torch.float16)
c = torch.empty_like(a)

cute.compile(
    ElementwiseAdd("vectorized"),
    from_dlpack(a, assumed_align=16),
    from_dlpack(b, assumed_align=16),
    from_dlpack(c, assumed_align=16),
)
PY

rg 'execve\(' /tmp/cute_dsl_probe/trace.log
rg -i 'ptxas|nvptxcompiler|libnvvm|libnvptx' /tmp/cute_dsl_probe/trace.log
```

## Answer

For `nvidia-cutlass-dsl[cu13]==4.4.2`, CuTe DSL does **not** put a standalone
`ptxas` executable on `PATH` for this path. The compiler-bearing native artifact
is the MLIR extension:

```text
.../nvidia_cutlass_dsl/python_packages/cutlass/_mlir/_mlir_libs/_cutlass_ir.cpython-312-aarch64-linux-gnu.so
```

Static ELF inspection showed:

- ELF: 64-bit AArch64 shared object, stripped
- size: `148,018,232` bytes in this environment
- dynamic dependencies: `libcuda.so.1` and system libraries
- no dynamic dependency on `libnvvm.so` or `libnvptxcompiler.so`

The extension contains NVIDIA compiler strings:

```text
nvPTXCompilerCreate(...)
nvPTXCompilerGetCompiledProgram(...)
Cuda compilation tools, release 13.1, V13.1.66
NVIDIA NVVM version 21.0.0
LLVM version 21.0.0git
CL-36699951
ptxas
```

A real CuTe DSL compile produced this PTX header:

```ptx
//
// Generated by NVIDIA NVVM Compiler
//
// Compiler Build ID: CL-36699951
// Cuda compilation tools, release 13.1, V13.1.66
// Based on NVVM 21.0.0
//

.version 9.1
.target sm_100a
.address_size 64
```

So the concrete versions for this package/environment are:

- CUDA compiler / embedded ptxas: `13.1`, `V13.1.66`
- compiler build ID: `CL-36699951`
- NVVM: `21.0.0`
- LLVM: `21.0.0git`
- emitted PTX ISA: `.version 9.1`
- emitted target on GB200: `.target sm_100a`

The generated CUBIN independently confirmed the ptxas metadata:

```text
Tool Name: ptxas
Tool Version: Cuda compilation tools, release 13.1, V13.1.66
Tool Command Line Arguments: -O 3 -arch sm_100a
CUDA Virtual SM: sm_100
CUDA Tool Kit Version: 13.1
```

The CUBIN also reported useful kernel/toolchain metadata:

```text
64-bit ELF: type=ET_EXEC, ABI=8, sm=100a, toolkit=13.1
EIATTR_REGCOUNT: 80
EIATTR_MERCURY_ISA_VERSION: 1.1
```

The runtime trace is the strongest evidence about how compilation happens.
During a real `cute.compile(...)`, `strace` showed only:

```text
execve(".../bin/python", ...)
execve("/sbin/ldconfig", ...)
```

There was no external `ptxas` process and no `openat` for `libnvvm.so`,
`libnvptxcompiler.so`, or a separate `ptxas` binary.

The practical model is therefore:

```text
CuTe DSL lowers Python DSL -> MLIR/NVVM in _cutlass_ir...so.
That same native extension embeds or statically links the NVIDIA
PTX compiler/assembler path.
It emits PTX and CUBINs annotated as produced by ptxas, but it does not
shell out to a separate ptxas executable for this package/env.
```

This distinction matters when debugging version mismatches: `which ptxas` may
find nothing, or may find an unrelated Torch/Triton assembler, while CuTe DSL is
using the compiler embedded in `_cutlass_ir...so`.
